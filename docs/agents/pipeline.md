# Pipeline Guide

`src/graph.py` is the canonical run order. Nodes mutate and return `AgentState`, which keeps the run observable and makes the final JSON report possible.

## Run Stages

1. `resolve_competitor`: normalizes the user input into a competitor profile.
2. `create_research_plan`: creates bounded category tasks.
3. `run_source_discovery`: calls tools from `src/tools/registry.py` for each category.
4. `review_source_coverage` and `detect_coverage_gaps`: score whether categories have enough source coverage.
5. `decide_next_step`: optional planner LLM call that can request one deeper search cycle.
6. `analyze_sources`: converts discovered source snippets into category-level analysis records.
7. `extract_evidence_claims`: creates grounded claims that later nodes can cite.
8. `generate_category_report_sections`: asks one subagent per research category to write the detailed report section with inline numeric citations.
9. `summarize_messaging` and `detect_recent_changes`: derive cross-source themes and changes.
10. `map_rippling_opportunities`: maps evidence to Rippling market opportunities. LLM runs use a dedicated validated mapper call with static Rippling positioning plus grounded competitor evidence; no-LLM runs and failed LLM calls use deterministic fallback mapping.
11. `generate_campaign_angles`: creates campaign recommendations from the evidence and mapped opportunities.
12. `evaluate_output`: scores grounding, coverage, caveats, and recommendation quality.
13. `write_outputs`: writes markdown, JSON, and the run log.

## Real-Source Mode

`state.real_sources_only` is enabled whenever an LLM client exists. In that mode, source discovery filters dummy tools out of the registry. Missing keys and API failures become gaps in the report instead of being filled with fake evidence.

This keeps LLM-generated reports from mixing real claims with placeholder sources. The no-LLM path can still use dummy data so tests and demos work offline.

## Data Flow

- `discovered_sources` contains raw public-source records and metadata.
- `tool_call_logs` contains observable API/tool calls, redacted requests, compact responses, and errors.
- `source_analyses` contains category summaries derived from sources.
- `extracted_claims` contains source-grounded claims with confidence.
- `category_report_sections` contains the detailed markdown generated by each category subagent.
- `rippling_opportunities` contains deterministic comparison opportunities for the late report subsection.
- `final_report` contains the structured report model that is serialized to JSON.

## Extension Points

- New source category: update schemas if needed, add tools to `src/tools/registry.py`, teach coverage/eval nodes what good coverage means, and add category report-section prompting.
- New API tool: subclass `BaseSourceTool`, return `ToolResult`, add redacted request/response logging, and cache expensive calls.
- New report section: prefer producing structured state first, then render in `output_writer`. Do not rely only on the final LLM to preserve required report structure.
- New Q&A behavior: update `src/nodes/report_qa.py`; it should answer from the saved report when possible and call Exa only for genuinely missing follow-up research.
